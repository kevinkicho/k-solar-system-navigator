HELIOSSolar System Navigator — physics & methodology

HELIOS is a physics-accurate interplanetary mission planner. Trajectories are computed against real Keplerian dynamics; the displayed Δv numbers are what an actual spacecraft would need.

Frame & ephemeris

Reference frameHeliocentric Ecliptic, J2000 (ICRF-aligned)
Planet positionsJPL "Approximate Positions of Major Planets" 1800–2050 — J2000 elements + per-Julian-century rates + great-inequality corrections (b·T² + c·cos(fT) + s·sin(fT) on L) for Jupiter–Neptune
Validated accuracyEarth at J2000: <5,000 km vs JPL DE; Mars opposition longitudes: 0.01°; Earth annual periodicity: 305 km
SpacecraftVoyager 1/2, Pioneer 10/11, New Horizons — heliocentric 2-body Kepler propagation from J2000 state vectors

Trajectory solver

MethodLambert's problem via universal-variable formulation
Root findingBracketed bisection on F(z) — convergence-safe across the full single-revolution regime
Branch selectionBoth short-arc and long-arc evaluated; cheaper Δv kept
ValidationEvery solution propagated back to target; rejected if miss > 1,000 km
Multi-leg routesLambert per leg; patched-conic gravity assist between legs
Planet-relative (same SOI)Parent-centered Lambert with Hohmann phase-window snap (e.g. Io→Ganymede ~2 d impulsive — correct for 2-body, not a multi-month tour). Earth→Moon analytic when 180° singular
High-energy Δv gateWarn above 30 km/s; fail only above 50 km/s (Mercury-class departures are real)

Gravity assist (patched-conic)

At each flyby the spacecraft's V∞ turning angle is checked against the maximum achievable at minimum-safe periapsis (planet radius × 1.1):

e = 1 + r_p · V∞² / μ_planet   ·   δ_max = 2·asin(1/e_min)

Infeasible swingbys are flagged TOO SHARP with the required-vs-minimum periapsis shown.

Vehicle model (concept-grade)

Results use a Need / Capability / Margin measurement triad. Sample architectures: Falcon 9 (illustrative payload-vs-C3 table), Starship + Super Heavy (legacy demo booster-only Δv, unrefueled LEO→TMI injection, N-tanker refuel), and abstract Δv budgets. Numbers are educational only — not SpaceX-certified, not flight operations.

Legacy demo mode keeps Super Heavy transfer Δv with Starship propellant reserved; cargo-aware arches use rocket-equation Starship capability at requested cargo mass. Margin may be Δv, cargo kg, or tankers depending on the primary metric.

Ephemeris fidelity badges

L1 defaultOffline JPL Approximate Positions for planning geometry. No network. Nominal error classes (λ, φ, ρ) from the JPL SSD table are shown on the Measurement Card — not formal covariances.
L2-compareAfter an explicit Horizons VECTOR compare only. Records that a Δr check was run; planning still uses L1 geometry. Not SPICE.
L2-planOptional offline sample-table endpoints for Lambert Need (educational DE-class samples when enabled). Animation still uses Approximate Positions. Not SPICE navigation.
L3 not a modeSPICE kernels / formal covariance / navigation solutions are not a planning mode. Educational path toward DE-class checks: L2-plan samples + optional Horizons compare. No .bsp kernels shipped.

JPL Approximate Positions accuracy table: ssd.jpl.nasa.gov/planets/approx_pos.html. Export records measurement.fidelity and methodology.ephemeris_backend.

Geographic sites (body-fixed endpoints)

Origin, destination, and optional flyby waypoints may use a geographic site: planetocentric latitude / east longitude / altitude above a reference surface. Coordinate system id: planetocentric+eastlon+h_above_ref.

Rocky bodiesMean or oblate ellipsoid radius; dual planetographic lat readout; optional planetographic input mode
Gas / ice giants1-bar / cloud-deck reference (no solid surface); high default parking altitude; System III–class longitude label
OrientationIAU-class W(t)=W₀+Ẇd (+ leading Moon/Mercury libration); ICRF pole α₀/δ₀ → ecliptic — concept-grade, not SPICE PCK
Share / exportSites in hash os/ds and mission JSON; multi-leg terminals and flyby geo sites when enabled

Launch geometry (educational)

Educational launch-site latitude bands (Cape / Vandenberg / Kourou class) gate asymptote |DLA| honesty. Parking-orbit Need uses circular parking at site altitude or body-kind default. Not range safety, not launch-window products, not flight design.

Multi-leg window search

Nearest-feasible and multi-leg window searches are local coarse or thorough seeds (workerized), not global mission optimization. Prefer denser “thorough” mode in Plan → Advanced when exploring multi-leg routes.

Known limitations

· 2-body Kepler propagation (no Jupiter/Saturn perturbations on the transfer orbit). Lambert solutions close to <2 km in 2-body; under full N-body the actual miss could be tens of km.

· Single-revolution Lambert. Type-II multi-revolution solutions are not yet a product mode.

· Visual inclinations are exaggerated 8× for legibility. All physics computations use real inclinations.

· Geographic body-fixed frames are concept-grade (IAU-class tables), not full SPICE kernels or survey geodesy.

· Planet-relative routes are parent-centered 2-body Lambert + parking sketches — not CR3BP, not low-energy lunar transfers, not tour design with resonant moon assists.

Optional Horizons comparison (educational)

Not SPICE. Not flight ops. Not required for planning. Trip planning always uses the offline JPL Approximate Positions model above. This stretch feature can optionally fetch a public NASA/JPL Horizons VECTOR table over the network and report distance error vs the approximate ephemeris for a selected major planet — for classroom curiosity only. No .bsp kernels, no covariance, no navigation solutions.
